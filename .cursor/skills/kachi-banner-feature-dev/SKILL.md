# kachi-banner-pro 機能開発スキル

## 前提

- **プロダクト**: `docs/product.md` を必ず読み、AI広告バナー生成Webアプリの目的・ユーザーストーリー・技術選定・実装方針を把握してから着手する。
- **役割**: シニアエンジニアとして、理解→設計→実装→レビューを一貫して行う。いきなりコードを書かず、GO が出るまで実装方針を提案に留める。

---

## 7フェーズの実行順序

次の順で進める。各フェーズの成果をユーザーに確認し、承認または指摘を受けてから次へ進む。

1. **要件の理解** … 何を実現するか・受け入れ条件・制約を整理する。
2. **コードベースの探索** … 既存の類似機能・ディレクトリ構成・型・API・状態管理を把握する。
3. **明確化質問** … 意思決定や判断が必要な場合は必ず質問する。曖昧な要件・エッジケース・トレードオフ・優先度についてユーザーに確認し、答えを得てから次に進む。**完全に不明点が解消されるまでは質問を繰り返す**。アーキテクチャ確定や実装着手前に曖昧な点が残っていると感じたら、追加で質問して解消する。
4. **アーキテクチャ設計** … 複数案を提示し、採用案を決める（ユーザーに選んでもらうか推奨を明示）。
5. **実装** … 設計に従い、既存規約に合わせて実装する。
6. **品質レビュー** … code-reviewer のやり方を参考に、複数観点（シンプルさ/DRY・バグ/正確性・プロジェクト規約）でレビューし、信頼度の高い指摘から対応する。必要なら修正する。
7. **サマリー** … 何をしたか・主要な決定・今後の推奨を短文でまとめる。

## プロダクト・技術の要点（docs/product.md 準拠）

- **目的**: LP URL + 参考バナーから、配信品質の広告バナーを簡単に生成する。静止画 1080x1080、プレビューは透かし付き、購入で透かしなし高画質。
- **技術**: フロント React+TS(Vite)、API Cloudflare Workers+Hono、画像 R2、DB Firestore、決済 Stripe、画像生成 OpenAI。Workers では Node 専用 API に依存しない。
- **方針**: ImageProvider 抽象化、preview と透かしなし原本の両保存、入力ハッシュでのキャッシュ、Agent モード時は方針提案→GO 後に実装。

## このプロジェクトで重視する実装方針

（過去のやりとりからユーザーが気にしがちな点を反映する）

- **最小限で始める**: 要件外の機能は削る。まず動く最小構成を実装し、必要に応じて拡張する。
- **設定の一元化**: モデル名・API URL・タイムアウトなどは `api/src/config.ts`（または shared）に集約し、散らかさない。
- **スキーマの共通化**: 型・空オブジェクト・パーサーは `shared/` に置き、api と web で共有する。プロンプト変更時は shared を 1 箇所直せばよい。
- **命名**: 略語より意図が分かる名前にする（例: `c` より `ctx`、`fetchImpl` は「渡された fetch」であることが分かるように）。
- **レイヤー**: controller / service / repository のように責務を分ける。ルートは HTTP だけ、ビジネスロジックは service、永続化は repository。
- **API 呼び出しの節約**: ランニングコストを抑えるため、不要な LLM 呼び出しや重複リクエストを避ける。編集はローカルで完結させ、保存時のみ API を叩くなど。
- **認証・ID**: 最初はダミーの userId / projectId で実装し、API の形（URL・JSON）を先に固める。後から Firebase Auth / Firestore を差し込んでも影響範囲を小さくする。
- **実装前の GO**: 方針や設計を提案したあと、ユーザーから GO をもらってからコードを書く。特に Agent モードでは「GO が出るまではコードを編集しない」。

## ADR の作成

- **タイミング**: 機能単位の実装が一区切りついたら、その機能について ADR を 1 本書く。
- **置き場所**: `docs/adr/` に `NNNN-短いタイトル.md` 形式で保存する。番号は **4 桁ゼロパディング**（0001, 0002, …）。例: `0001-lp-analysis-api-design.md`。
- **内容**: 次の項目を簡潔に書く。
  - **タイトル・日付**
  - **状況**: 何を決める必要があったか。
  - **決定**: 採用した方針（型・エンドポイント・ディレクトリ・技術選びなど）。
  - **理由**: なぜその選択にしたか（トレードオフ・制約）。
  - **結果**: その決定により何ができたか・残っている課題。

後続のタスクで「なぜこうなっているか」を参照するときは、この ADR を読む。

## フェーズごとのチェックリスト

### 1. 要件の理解
- [ ] `docs/product.md` を読んだ。
- [ ] 依頼された機能の目的・受け入れ条件・制約を 1 文〜数文でまとめた。
- [ ] 不明点をリストアップした（次の「明確化質問」で使う）。

### 2. コードベースの探索
- [ ] 関連しそうなディレクトリ・ファイルを特定した（api/, web/, shared/）。
- [ ] 既存の型・API・状態管理（Jotai / TanStack Query）の使い方を把握した。
- [ ] 類似機能の実装パターンをメモした。

### 3. 明確化質問
- [ ] 意思決定・判断が必要な箇所を洗い出し、必ず質問した（曖昧な要件・エッジケース・優先度・トレードオフ）。
- [ ] ユーザーの回答または「お任せ」を確認した。
- [ ] 不明点が残っていないか確認し、あれば追加で質問して解消してから次フェーズに進んだ。

### 4. アーキテクチャ設計
- [ ] 2 つ以上の案（または 1 案 + 代替の簡略版）を提示した。
- [ ] トレードオフと推奨を明示した。
- [ ] ユーザーが採用案を選ぶか、推奨で進めることを確認した。

### 5. 実装
- [ ] 採用した設計に従った。
- [ ] 既存の命名・config/shared の使い方・レイヤーに合わせた。
- [ ] 必要なら TodoWrite で進捗を管理した。

### 6. 品質レビュー
- [ ] code-reviewer 風に、シンプルさ/DRY・バグ/正確性・プロジェクト規約の各観点でレビューした。信頼度の高い指摘を優先して報告した。
- [ ] 指摘があれば修正するか「後回し」をユーザーに確認した。

### 7. サマリー
- [ ] 何を実装したか・主要な決定・今後の推奨を短文でまとめた。
- [ ] 機能単位で ADR を `docs/adr/NNNN-タイトル.md` に作成した（番号は 4 桁ゼロパディング）。

## スキルの更新

ユーザーからの **レビュー指摘**・**明確化質問への回答**・**判断や方針の指示** は、今後の実装に活かすため、本スキルを更新する。

- **実装方針の追加・修正**: 「このプロジェクトで重視する実装方針」に、ユーザーが繰り返し指摘したり明示したルールを追記する。
- **reference.md の更新**: フローやテンプレートの変更、よくある判断のメモなどがあれば反映する。
- 更新後は「スキルをこう更新した」と一言サマリーする。
