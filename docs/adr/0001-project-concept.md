# 0001. プロジェクト概念の導入（一覧・作成・名前編集）

**日付**: 2025-03-02

## 状況

バナー生成を「商材ごと」に管理・再利用できるようにするため、プロジェクトという概念を導入する必要があった。LP 分析コンテキストをプロジェクトに紐づけて保存し、複数商材を毎週扱うユースケースでアクセスしやすい UI にしたい。バナー生成機能の実装は今回のスコープ外。

## 決定

- **API**
  - **一覧**: `GET /projects` は **id と name のみ**返す（`ProjectListItem[]`）。サイドバー表示を軽くし、表示を遅くしない。
  - **詳細**: `GET /project/:id` で full `Project`（lpUrl, lpStructuredContext 等）を取得。一覧の項目をタップしたときにのみ呼ぶ。
  - **作成**: `POST /project`。LP 分析完了時に name / lpUrl / lpStructuredContext / lpRawAnalysisMarkdown を送り、作成されたプロジェクトの id で `/project/:id` に遷移。
  - **名前更新**: `PATCH /project/:id` で `{ name }` を送り、プロジェクト名を後から編集可能に。
  - 永続化は **インメモリ**（Map<projectId, Project>）。userId は固定のダミー。後から Firestore に差し替え可能な形にした。

- **フロント**
  - `/new` の Step1 で **プロジェクト名**と **LP URL** を入力。分析完了時に `POST /project` でプロジェクト作成 → `navigate(/project/:id)`。
  - サイドバーは「ダッシュボード」の下に「プロジェクト」セクションを設け、`GET /projects` の結果（id, name）を列挙。タップで `/project/:id` へ遷移。
  - プロジェクト詳細でプロジェクト名を編集する UI を追加し、`PATCH /project/:id` で保存。保存後に一覧を invalidate してサイドバーを更新。

## 理由

- 一覧と詳細でレスポンスを分けたのは、サイドバーの表示速度を優先するため。一覧は id/name のみにし、詳細はタップ時のみ取得する。
- プロジェクト名はユーザー入力のみ（LP 分析完了時にその名前で保存）。コストを増やさず、後から編集可能にした。
- 永続化をインメモリにしたのは、API の形（URL・JSON）を先に固め、後から Firebase/Firestore を差し込む方針に合わせるため。

## 結果

- 新規フロー: プロジェクト名＋LP URL 入力 → LP 分析 → プロジェクト作成 → 詳細画面へ遷移。
- サイドバーでプロジェクト一覧（名前のみ）を表示し、タップで詳細を開ける。
- 詳細画面でプロジェクト名を編集し、一覧に反映される。
- 存在しない id で `/project/:id` にアクセスした場合は「プロジェクトが見つかりません」を表示し、ダッシュボード／新規作成へ誘導。
- 今後の課題: Firestore への移行、バナー生成機能の実装。
